# 拆书结果可理解性评审与改造策略

日期：2026-06-25

## 问题定义

当前拆书能力已经能产出很多东西：关系图谱、角色卡、世界书、大事纪、章节功能、伏笔表、情绪节奏、续写包、导出资产。

但用户真正遇到的问题不是“没有信息”，而是：

```text
信息很多，但不好理解。
图谱能看见节点和线，却看不懂这本书为什么这样设计。
卡片看起来像资料库，不像一个人帮我讲清楚这本书。
大事纪只是事件列表，不像思维导图式的故事理解路线。
```

这说明当前产品还缺一层：

```text
拆书理解层
```

也就是把抽取结果重新组织成作者能顺着看的结构：主线、阶段、人物弧线、关系变化、伏笔回收、情绪曲线和可学习动作。

## 专业评审团

### 1. 资深网文编辑

现在的拆书结果像“资料员整理出来的素材”，不是“编辑给作者讲这本书为什么能留住读者”。

关系图谱只告诉用户“谁和谁有关”，但作者真正想知道的是：

- 主角被谁压迫？
- 谁提供目标、代价、诱惑或阻碍？
- 哪段关系制造追读？
- 哪个关系在第几阶段发生变化？
- 这个关系变化对爽点、虐点、甜点或悬念有什么作用？

如果图谱只显示节点和边，而不解释“关系在剧情里干什么”，作者很难学到东西。

### 2. 叙事结构分析师

“大事纪”这个方向是对的，但现在的大事纪更像流水账。

对作者有用的不是所有事件，而是事件之间的因果和功能：

```text
触发事件 -> 压力升级 -> 主角选择 -> 代价出现 -> 关系变化 -> 阶段兑现 -> 新钩子
```

时间线应该至少分成三层：

- 主线阶段：故事从哪里开始，经历几次升级，到哪里兑现。
- 人物变化：主角、反派、关键关系在每一阶段发生什么变化。
- 读者期待：每一阶段给读者什么承诺，兑现了什么，又留下什么没兑现。

单纯列事件，没有形成“为什么下一章还要看”的路径。

### 3. 知识可视化设计师

当前关系图谱更接近知识图谱，不是阅读理解图。

知识图谱的问题是：节点越多，用户越迷路。对作者来说，第一层应该不是全量图，而是“导览图”。

建议改成四种视图：

1. 总览地图：一句话卖点在中间，向外分出主角、压力来源、核心关系、世界规则、爽点机制、悬念机制。
2. 主线时间轴：按阶段讲清楚事件因果。
3. 关系变化图：只看 5-8 条最重要关系，每条关系显示“起点 -> 转折 -> 当前状态”。
4. 伏笔回收表：每个未完成期待对应“埋下、推进、回收、风险”。

全量图谱应该放到第二层，作为查证和探索工具，不应该是用户理解一本书的第一入口。

### 4. UX 研究员

用户第一次看到拆书结果时，不应该先面对一堆卡片。

正确顺序应该是：

```text
先给结论
再给路线
再给细节
最后给证据和导出
```

建议拆书结果首页改成：

1. 这本书靠什么吸引读者。
2. 它的故事是怎么一步步升级的。
3. 哪几个人物关系最关键。
4. 哪些伏笔和未完成期待支撑追读。
5. 作者可以学什么，不能照搬什么。

如果用户愿意深入，再打开角色卡、世界书、完整图谱和导出包。

### 5. AI 抽取架构师

后端现在已经有不少字段能支撑改造：

- `chapterFunctionTable`
- `foreshadowingLedger`
- `emotionalBeatMap`
- `pacingCurve`
- `readerPromiseChecklist`
- `conflictMatrix`
- `relationships`
- `chronicle`
- `plotlines`

但这些字段目前是并列资产，没有被综合成理解模型。

建议新增一个派生层，不一定一开始就改模型 schema，可以先在前端或后端从已有字段生成：

```text
BookComprehensionMap
```

它不追求信息最全，而追求最容易看懂。

### 6. 内容产品负责人

拆书模块的产品目标需要重新表述：

当前偏向：

```text
把一本书拆成很多可导出资产。
```

应该转向：

```text
把一本书讲清楚：它为什么能留住读者，作者能学什么，下一本书怎么用。
```

导出资产仍然重要，但它应该是理解之后的结果，不是第一体验。

## 核心评判

当前拆书模块最大问题不是能力不足，而是表达形态错位。

```text
现在展示的是“抽取结果”。
用户需要的是“理解路线”。
```

关系图谱、角色卡、大事纪、世界书这些都应该保留，但它们需要被重新组织成：

- 先总览。
- 再按故事阶段展开。
- 再解释人物关系为什么重要。
- 再落到可学习规则。
- 最后才是完整资料和导出。

## 推荐解决策略

### 策略一：新增“拆书导览页”，代替直接进入全量图谱

拆书完成后，默认进入“拆书导览”，而不是直接进入图谱或卡片堆。

页面结构建议：

```text
这本书一句话
-> 核心吸引力 3 条
-> 主线升级地图
-> 关键人物关系
-> 伏笔与未完成期待
-> 作者可学习动作
-> 不能照搬清单
```

示例文案：

```text
这本书不是靠设定多留住读者，而是靠“身份压力 + 关系误解 + 阶段性反击”持续制造追读。
```

### 策略二：把“大事纪”升级成“故事阶段时间轴”

不要只列事件，要把事件整理成阶段。

建议结构：

| 字段 | 作用 |
| --- | --- |
| 阶段名 | 例如“被压迫”“发现机会”“第一次反击”“更大危机” |
| 章节范围 | 这个阶段覆盖哪些章节 |
| 阶段目标 | 主角或叙事在这个阶段要解决什么 |
| 关键事件 | 只放 2-4 个真正推动故事的事件 |
| 关系变化 | 哪些人物关系因此改变 |
| 读者期待 | 这一阶段让读者继续看的理由 |
| 阶段钩子 | 阶段末尾留下什么未完成问题 |
| 可学习点 | 作者写同类故事时可以学什么 |

它应该看起来像：

```text
阶段 1：主角被压到必须反击
第 1-3 章
触发：身份被否定
升级：关键资源被夺
选择：主角冒险寻找翻盘证据
关系变化：主角与导师从试探变成交易
读者期待：他能不能证明自己被冤枉？
可学习点：开局不要只写惨，要写“惨之后必须行动”。
```

### 策略三：把关系图谱分成“理解版”和“探索版”

当前全量图谱可以保留为探索版。

新增理解版关系图，只显示最重要的 5-8 条关系，并且每条关系要讲清楚：

- 关系双方。
- 当前关系状态。
- 关系在故事里的功能。
- 第一次出现在哪。
- 哪个事件改变了关系。
- 这条关系制造了什么读者期待。
- 作者可学习的关系写法。

示例：

```text
主角 -> 导师
表面关系：交易/试探
故事功能：给主角一条翻盘路径，但不直接替他解决问题。
变化节点：主角拿出证据后，导师从旁观变成有限支持。
读者期待：导师到底知道多少隐藏真相？
可学习点：导师不要一开始全知全能，要先作为门槛和压力存在。
```

### 策略四：把卡片从“资料卡”改成“理解卡”

目前很多卡片的问题是只有描述，没有解释。

每张卡片建议固定为五段：

```text
它是什么
它在故事里干什么
它为什么能留住读者
作者可以怎么学
哪些地方不能照搬
```

角色卡示例：

```text
角色：导师型压力源
它是什么：一个掌握资源但不轻易帮主角的人。
故事功能：给主角设门槛，让主角必须证明自己。
读者作用：读者会期待主角什么时候获得认可。
可学习点：导师越强，越不能免费送外挂；必须让主角付出代价。
不要照搬：原作姓名、身份、专属组织、标志台词和具体事件。
```

### 策略五：新增“思维导图式结构”

不一定要一开始做复杂可视化，先用结构化树就能解决 70% 问题。

建议导览图结构：

```text
核心读者承诺
├─ 主角为什么必须行动
│  ├─ 压力来源
│  ├─ 代价
│  └─ 翻盘机会
├─ 故事怎么升级
│  ├─ 第一次冲突
│  ├─ 第一次反击
│  ├─ 更大压力
│  └─ 阶段兑现
├─ 人物关系怎么推动追读
│  ├─ 压迫关系
│  ├─ 交易关系
│  ├─ 误解关系
│  └─ 暧昧/羁绊关系
├─ 伏笔怎么留住读者
│  ├─ 已埋伏笔
│  ├─ 部分回收
│  └─ 未回收风险
└─ 作者可以学什么
   ├─ 开局承诺
   ├─ 冲突升级
   ├─ 关系钩子
   └─ 章末钩子
```

这比一上来展示完整力导向图更容易读懂。

### 策略六：导出报告分成“阅读版”和“资料版”

当前导出更像资料包。建议增加一个默认导出：

```text
拆书阅读报告
```

它只服务阅读理解，不服务导入工具。

报告结构：

1. 这本书为什么能吸引读者。
2. 故事阶段地图。
3. 关键关系变化。
4. 伏笔和读者期待。
5. 可以学习的写法。
6. 不能照搬的内容。
7. 附录：完整角色卡、世界书、图谱 JSON。

原有世界书、角色卡、Tavern 卡、JSON 都放到资料版导出。

## 建议的数据结构

可以先新增派生结构，不破坏现有拆书结果：

```ts
interface BookComprehensionMap {
  headline: string;
  corePromise: string;
  whyItWorks: string[];
  readingPath: Array<{
    phase: string;
    chapterRange: string;
    trigger: string;
    escalation: string;
    choiceOrTurn: string;
    payoff: string;
    openHook: string;
    readerReasonToContinue: string;
    learnableMove: string;
  }>;
  keyRelationships: Array<{
    from: string;
    to: string;
    state: string;
    storyFunction: string;
    changeMoment: string;
    readerExpectation: string;
    learnableMove: string;
    evidence: string[];
  }>;
  promiseMap: Array<{
    promise: string;
    setup: string;
    progress: string;
    payoffOrRisk: string;
  }>;
  beginnerTakeaways: Array<{
    rule: string;
    why: string;
    howToUse: string;
    avoid: string;
  }>;
}
```

第一版可以从已有字段派生：

- `headline` 来自 `book.oneSentencePremise` + `coreAppeal`。
- `readingPath` 来自 `chronicle` + `chapterFunctionTable` + `pacingCurve`。
- `keyRelationships` 来自 `relationships.edges` + `conflictMatrix`。
- `promiseMap` 来自 `readerPromiseChecklist` + `foreshadowingLedger`。
- `beginnerTakeaways` 来自 `plotlines.reusablePattern` + `qualityDiagnosis.strengths`。

## 前端改造方案

### 新增结果首页：拆书导览

位置：`/book` 拆书任务完成后默认展示。

模块：

- 顶部一句话：这本书靠什么留住读者。
- 三张核心卡：主卖点、主冲突、主钩子。
- 故事阶段时间轴。
- 关键关系变化。
- 伏笔与未完成期待。
- 可学习规则。
- 按钮：查看完整图谱、查看角色卡、导出阅读报告。

### 页面展示规划：先读懂，再探索

第一屏不再直接展示全量关系图、角色卡和资产包，而是按作者理解顺序组织：

1. `拆书导览`：一句话说明这本书靠什么留住读者。
2. `推荐阅读路径`：用锚点告诉用户先读懂、再查证据、学写法、看人物世界、探索图谱，避免一上来迷失在资产堆里。
3. `理解版思维导图`：以核心承诺为中心，向外展开主角压力、关系钩子、冲突升级、伏笔期待和可学写法。
4. `故事阶段时间轴`：把章节功能改写成触发、升级、钩子和读者继续读的理由。
5. `关键关系故事线`：按首次出现章节解释关系如何进入故事、制造什么读者问题、作者能学哪种关系写法。
6. `关键关系为什么重要`：先解释关系的故事功能，再允许进入完整关系图谱。
7. `伏笔和未完成期待`：只展示会影响追读的问题，不把全部资料一次性铺开。
8. `作者可以先学这几招`：把拆书结果转换成可迁移的写法。
9. `把这次拆书结果用于新书`：复制原创开局 Prompt，要求换掉原作姓名、专有名词、关系网和标志桥段。

这个顺序的产品目标是让用户先获得“我看懂了”的确定感，再进入图谱、角色卡、世界书、导出包等深水区。

### 导出区规划：按任务分组，不按格式堆叠

导出入口不应该让用户先理解 Markdown、JSON、World Info、角色卡这些格式差异，而应该先回答“我要拿这份拆书结果做什么”。

推荐分三组：

1. `先读懂`：拆书阅读报告、完整学习报告、避险清单。面向作者复盘和团队讨论。
2. `继续创作`：卷纲/大纲、改写提示词包、续写数据包、风格说明书。面向新书规划和 AI 辅助创作。
3. `资料归档 / 工具导入`：完整 JSON、角色卡、世界书、SillyTavern 世界信息。面向自动化、外部工具和高级用户。

每组只标一个首选入口，避免所有按钮都显示“推荐”导致推荐失效。第一阶段首选分别是：拆书阅读报告、卷纲/大纲、完整 JSON。

### 弱数据兜底：像老师一样告诉用户怎么补

拆书结果缺关系、缺时间轴、缺伏笔时，不应该只显示“暂无数据”。作者真正需要知道的是：

1. 这块为什么缺。
2. 这会影响什么判断。
3. 下一次应该补什么材料。
4. 当前页面还有哪个结果可以先看。

空状态统一使用“原因 + 下一步动作”的结构。例如：

- 关系不足：提示补主角与反派、导师、盟友、交易对象的直接互动段落。
- 时间轴不足：提示保留章节标题，上传连续章节而不是零散片段。
- 伏笔不足：提示检查前三章是否有未解决问题、秘密、承诺、威胁或反击机会。
- 图谱为空：提示补角色别名表，避免同一角色被拆成多个节点。

这类兜底的目标不是遮丑，而是把模型抽取失败转成作者能执行的补料动作。

### Dashboard 规划：从指标墙变成复诊教练

Dashboard 不应该只是任务列表或指标墙。它应该回答作者三个问题：

1. 最近质量是在变好、持平还是变差。
2. 常见问题是否在重复出现。
3. 下一轮应该先改正文、改 Prompt，还是沉淀成方法论。

页面结构建议：

- 顶部保留复诊次数、分数变化、Prompt 有效率、方法论卡数量。
- 指标下方增加 `编辑建议`：用 Gate、分数变化、常见问题、Prompt 归因和方法论卡生成下一步动作。
- 质量趋势只解释方向，不把分数当成绝对真理。
- Prompt 有效率必须标注“相关性观察，不是因果证明”。
- 常见问题和方法论卡要能直接指导下一轮 Prompt 或写前自查。
- 项目 Markdown 导出必须包含 `编辑建议`，让离线文档也能沉淀下一轮动作，而不是只保存历史记录。

这样 Dashboard 的定位才是“复诊教练”，而不是历史任务列表。

### 图谱页改造

保留现有力导向、社区、时间线布局，但新增“解释面板”：

点击边时，不只显示关系名和证据，还显示：

- 这条关系的故事功能。
- 它制造的读者期待。
- 它在第几个阶段变化。
- 作者可学习的关系写法。

### 卡片页改造

卡片不再只按资产类型排列，而按读者理解顺序排列：

```text
主线卡 -> 关系卡 -> 伏笔卡 -> 世界规则卡 -> 写作规则卡 -> 原创化避险卡
```

每张卡都必须有“作者可以怎么用”。

## 后端改造方案

### P0：不改模型，先派生理解层

新增一个纯函数：

```text
buildBookComprehensionMap(result)
```

从现有字段生成导览数据，先解决展示问题。

优点：

- 风险低。
- 不影响现有导出格式。
- 可以快速迭代前端体验。

### P1：增加二次总结 Prompt

在 reduce 完成后，额外做一次“编辑导览总结”：

```text
你不是资料整理员，而是拆书老师。
请把以下拆书资产重组为作者能看懂的阅读路线图。
不要列全量信息，只保留最能解释这本书为什么留住读者的内容。
```

输出 `BookComprehensionMap`。

### P2：让 map 阶段补更适合导览的数据

章节 map 阶段增加字段：

- 本章读者为什么继续读。
- 本章改变了哪段关系。
- 本章新增/推进/兑现了什么承诺。
- 本章属于哪个故事阶段。

这样后续时间线和关系变化会更稳。

## 优先级建议

### 第一阶段：先让结果“看得懂”

目标：拆书完成后，用户能在 1 分钟内明白这本书的核心结构。

任务：
