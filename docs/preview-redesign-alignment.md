# 预览页设计与现有工作区功能对齐

日期：2026-07-12

## 结论

`ai_novel_diagnosis_redesign.html` 的左侧目录应该调整。

原因不是视觉问题，而是信息架构归属问题：章节目录、改稿复诊、历史记录、方法论卡片等能力已经在项目工作区或书籍工作区实现。快速诊断页的左侧如果继续展示这些内容，会让用户误以为它们是快速诊断页的一部分，也会让后续落地时出现重复状态、重复路由和重复组件。

目标形态：

- 全局左侧保留工作区入口：诊断、我的书籍、研究、设置；二级能力默认收起。
- 当前页面内部只展示当前任务阶段：输入稿件、查看诊断、复制修改指令。
- 章节目录、意见处理、版本、复诊、方法论和导出回到项目/书籍工作区。
- 深度质检作为诊断工作区里的独立进阶任务，用 stepper 表达“参考样本 -> 评分标准 -> 评分结果”。
- 首页/快速诊断页必须把快速诊断放在第一任务；深度质检、我的书籍、研究和 AI 设置通过左侧导航与模型状态卡进入。

完整功能取舍、保留/完善/后置/砍独立入口的判定，见：

- [功能取舍与页面位置预留](./feature-prioritization-and-design-slots.md)

## 当前导航事实

原项目已经有四工作区导航事实源：

- `apps/web/src/lib/workspace-routes.ts`
- `apps/web/src/stores/workspace-nav-store.ts`
- `apps/web/src/components/workspace/WorkspaceNav.tsx`

现有文档也已经明确：

- `docs/ux-refactor/02-information-architecture.md` 要求左栏只回答“我在哪个任务”。
- 任务有前置依赖时使用 stepper，而不是把步骤伪装成平级导航。
- 同一内容只能有一个主要归属；其他入口只能链接过去，不能复制实现。

## 静态预览页调整记录

已调整：

- `ai_novel_book_preview_v18_visual_polish/ai_novel_diagnosis_redesign.html`

调整内容：

- 左侧从“诊断/项目/研究”的功能清单，收敛为“工作区 / 进阶入口 / 模型状态”。
- “历史记录、方法论卡片、改稿复诊”等不再作为快速诊断页常驻目录项。
- 不在 UI 中展示功能归属分组；复诊记录、方法论库、诊断看板、导出资产进入我的书籍后出现，拆书图谱、样本研究、历史任务进入研究工作区后出现。
- 深度质检入口链接到 `deep_quality.html`，继续作为进阶诊断任务。
- 移除首页主区的四个功能卡：快速诊断、深度质检、导入整本书籍、AI 设置。
- 模型连接状态回到左下角，作为全局状态位；模型卡下方保留 AI 设置快捷按钮。
- “输入稿件 / 查看诊断 / 复制修改指令”保留为主区步骤条，不再放进左侧导航。
- 示例入口统一为“载入示例”，避免顶部“载入示例”和输入卡片“填充示例”重复。
- 移除首页主区的“功能去向和成熟度”能力地图；能力成熟度只作为左侧折叠归属和文档判定存在。
- 对比 `/diagnose/quick` 现状后，主区采用更工具化的表达：标题收敛为“快速诊断”，副文案说明“为什么留不住读者、下一版先改哪里”，右侧卡片改为“你将得到”，列出最大流失点、正文证据、三步修改方案和下一轮修改指令。

## 建议的新目录模型

### 全局工作区

| 工作区 | 主入口 | 说明 |
| --- | --- | --- |
| 诊断 | 快速诊断 | 新手主路径：贴章节 -> 找最大流失点 -> 复制修改指令 |
| 我的书籍 | 书籍列表/当前项目 | 管理章节、诊断意见、版本、改稿计划、复诊和导出 |
| 研究 | 整书拆解 | 上传整本、章节切分、拆书、样本对比和研究资料 |
| 设置 | 模型设置 | AI provider、隐私、本地连接状态 |

### 诊断工作区内部

| 任务 | 展示方式 | 备注 |
| --- | --- | --- |
| 快速诊断 | 单页任务流 | 输入、报告、修改指令是状态，不是左侧一级目录 |
| 深度质检 | 独立页 + stepper | 参考样本、评分标准、评分结果 |
| 证据链 | 结果内详情/Tab | 有结果后出现，不常驻左侧 |

### 项目工作区内部

| 任务 | 展示方式 | 备注 |
| --- | --- | --- |
| 书籍/章节 | 项目页或章节工作区 | 章节树、批注、意见状态都放在这里 |
| 复诊记录 | 独立列表/详情 | 从快速诊断或章节页跳入 |
| 方法论卡片 | 独立页面 | 支持沉淀、筛选、详情 |
| 导出资产 | 独立页面或项目操作 | 依赖当前项目已有资产 |

## 后续设计落实清单

### 1. 对齐 Next 代码的左侧导航

当前 `apps/web/src/components/workspace/RedesignWorkspaceShell.tsx` 仍有重复导航倾向：

- `证据对照` 常驻在诊断组中；
- `历史记录` 文案指向项目概览；
- `深度质检` 和 `证据对照` 仍指向旧的 `/diagnose/chapter` 风格入口。

建议下一步：

- 将全局左侧收敛为四个工作区入口。
- 将快速诊断页面内的“输入稿件 / 查看诊断 / 复制修改指令”保留为页面内步骤条。
- 将深度质检页面内的“参考资料 / 评分标准 / 评分结果”保留在 `DeepDiagnosisStepper`。
- 把证据链从全局导航迁移为报告里的详情入口。

### 2. 统一路由文案

需要同步检查：

- `workspace-routes.ts`
- `WorkspaceNav.tsx`
- `RedesignWorkspaceShell.tsx`
- `apps/web/README.md`
- `docs/ux-refactor/02-information-architecture.md`

建议把用户可见文案统一为：

- 诊断
- 我的书籍
- 研究
- 设置

其中“项目”可以在代码层继续叫 `project`，但面向作者的 UI 更建议叫“我的书籍”。

### 3. 快速诊断页后续设计方向

保留当前主任务：

- 输入正文；
- 选择题材和本次关注点；
- 运行诊断；
- 展示最大流失点、证据、修改优先级；
- 复制修改指令；
- 进入书籍/章节工作区继续改稿复诊。
- 首屏展示模型连接状态和 AI 设置入口，避免用户开始诊断前找不到模型设置。
- 首屏展示进阶但不抢主任务的入口：深度质检、导入整本书籍。

需要避免：

- 在左侧复刻章节目录；
- 在右侧常驻完整历史；
- 在快速诊断页直接展开方法论库；
- 把深度质检、证据链、复诊做成同级主导航。

### 4. 深度质检页后续设计方向

静态原型 `deep_quality.html` 已经按“参考拆书 -> 标准库 -> 整书质检 -> 报告落地”补齐一版。

后续落地到 Next 时，应对应：

- `DeepDiagnosisStepper` 的参考资料、rubric、score；
- `research-library.ts` 的证据链构建；
- `workspace-store.ts` 的 rubric/score 结果状态；
- 项目工作区的改稿计划或章节意见状态。

## 验收标准

- 左侧导航不再出现同一能力的重复归属。
- 快速诊断首屏只服务一个主任务：贴章节并诊断。
- 诊断结果生成后，主区优先展示报告和修改指令。
- “进入书籍工作区”能承接章节、意见、版本、复诊和导出。
- 深度质检的步骤不和全局导航混用。
- 移动端只有单列任务流，不出现双层常驻导航。

## 维护约定

后续任何新的页面设计稿，需要先写清：

1. 它属于哪个工作区。
2. 它是任务、步骤、Tab、详情检查器，还是设置页。
3. 它复用哪些现有代码能力。
4. 它新增哪些状态或数据。
5. 它是否会复制已有工作区能力；如果会，必须改成链接或引用。
