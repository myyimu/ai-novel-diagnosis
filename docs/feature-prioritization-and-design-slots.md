# 功能取舍与页面位置预留

日期：2026-07-12

> 功能优先级服从 [`product-doctrine.md`](./product-doctrine.md)：优先级不按“分析指标多少”判断，而按它是否蒸馏编辑方法并进入证据、作者确认、真实改稿和复诊闭环判断。

## 产品纪律

AI 网文诊断台的第一层只服务一个问题：

```text
编辑会先检查哪里，证据是什么，下一版先改什么。
```

所有功能都必须围绕这条链路归位：

```text
快速诊断 -> 原文证据与替代解释 -> 作者确认 -> 修改计划 -> 保存新版本 -> 独立复诊 -> 经确认的规则沉淀
```

进阶能力可以保留，但不能抢首页第一注意力。首页允许被看见的功能只有两类：

- 支撑第一次诊断的必要能力：示例、模型状态、开始诊断。
- 必要的全局入口：左侧导航里的我的书籍、研究、设置。

注意：

- 模型状态是全局工具状态，适合放在左下角；模型名称下应保留 AI 设置快捷按钮，因为配置和连接状态属于同一件事。
- “输入稿件 -> 查看诊断 -> 复制修改指令”是当前页面的步骤，不是左侧导航项；左侧导航只放工作区或任务入口。
- 功能归属是设计和路由原则，不是用户导航；不要在首页左侧放“我的书籍内 / 研究工作区内 / 结果内出现”这类元信息。

## 功能取舍总表

| 功能 | 决策 | 页面位置 | 为什么 | 下一步 |
| --- | --- | --- | --- | --- |
| 诊断台 / 快速诊断 | 核心保留，继续强化 | 首页第一屏主任务、`/diagnose/quick` | 低门槛复现编辑初筛过程 | 强化证据锚点、证据充分度、替代解释、作者确认和“进入改稿” CTA |
| 修改指令 / 改稿 Prompt | 核心保留，继续强化 | 诊断结果主区 | 直接把诊断变成行动，是和 AI 直接写的差异 | 增加保守修、强冲突、重写开头等模式，但默认只给一个推荐 |
| 复诊记录 | 保留，必须完善 | 我的书籍 / 项目工作区、`/project/revisions` | 闭环差异点；判断改稿是否真的有效 | 保存 V1/V2 正文、作者采用项、问题状态、独立复核和下一轮动作 |
| 方法论库 | 保留，但改表达 | 我的书籍 / 项目工作区、`/project/methodology` | 长期价值；把经确认的编辑经验沉淀成写前自查规则 | 页面内解释为“经验证的常犯问题和改稿规则”，一次模型输出不得自动入库 |
| AI 设置 | 保留，必要支撑 | 设置工作区、首页轻入口、`/settings/provider` | 模型可用性是信任前提 | 首页只显示连接状态和配置入口，不展开复杂表单 |
| 诊断看板 | 保留，但迁移定位 | 项目工作区或诊断结果汇总，不放设置第一层 | 它应该是复诊教练，不是指标墙 | 优先显示问题解决率、采纳/误报、覆盖率和下一轮建议，分数退居兼容信息 |
| 深度质检 | 保留，进阶强化 | 诊断工作区进阶入口、`/diagnose/deep` | 严肃作者需要成熟样本标准和证据链检查 | 首屏说明“急诊跑通后再用”，用 stepper 承接参考 -> 标准 -> 证据化检查，评分退居次要 |
| 拆书图谱 | 保留，后置为研究卖点 | 研究工作区、整书拆解结果内 | 差异化强，但会增加新手理解成本 | 默认先展示拆书导览和关键关系故事线，全量图谱作为探索层 |
| 样本研究 / 样本对比 | 保留，后置 | 研究工作区、`/research/compare` | 用成熟作品学习结构有价值，但依赖已有样本 | 空状态引导先导入整本；多书对比不作为首页卖点 |
| 历史任务 | 保留，归到研究任务恢复 | 研究工作区的任务列表或历史子页 | 长任务恢复必要，但不是设置项 | 从设置迁出；只展示可继续、失败、已完成任务 |
| 导出资产 | 保留，但收缩默认格式 | 我的书籍 / 项目工作区、`/project/export` | 用户能带走成果，适合项目结束和复盘 | 按“先读懂 / 继续创作 / 资料归档”分组，不按格式堆按钮 |
| 证据对照 / 证据链 | 保留为结果详情，不做全局入口 | 诊断报告内 Tab/详情检查器 | 有结果才有意义，常驻导航会制造空页面 | 从全局左侧移除，作为报告内对象详情 |
| 评分报告 | 保留为深度质检结果态，不做独立主入口 | `/diagnose/deep` 内结果步骤或兼容路由 | 与深度质检高度重叠 | 旧路由保留兼容，新导航不扩散 |
| 爆款套路库 / 套路库 | 合并下沉 | 研究工作区“样本学习/规律沉淀” | 名称容易低质化，偏离 AI 编辑台定位 | 不单独作为一级页面；改名为“样本规律” |
| 研究库问答 | 暂停主推 | 研究工作区高级能力 | 容易变泛聊天，削弱证据链定位 | 只允许基于上传样本证据回答结构问题 |
| 多书对比 | 暂停主推 | 研究工作区高级能力 | 需要足够样本后才有价值 | 样本不足时不展示主 CTA |
| Auth/User/Common 模板 | 暂不产品化 | 不进入用户页面 | 当前默认本地单人使用 | 保留工程骨架，不出现在产品文案 |
| `/workspace`、`/starter` 等旧入口 | 保留兼容，砍掉推广 | 旧链接重定向或兼容页 | 避免旧链接断，但不应继续扩大心智 | 文档标记为 legacy |

## 分层结论

### P0：必须保留并持续打磨

- 诊断台 / 快速诊断
- 原文证据
- 修改指令
- 复诊记录
- AI 设置

P0 的“复诊”只有在保存真实正文版本、作者采用/拒绝决定和 issue 状态变化后才算完成；诊断历史和分数趋势不能替代它。

这些构成产品最短闭环。任何首页设计都必须优先服务它们。

### P1：保留并完善，作为第二层价值

- 方法论库
- 诊断看板
- 深度质检
- 导出资产

这些是用户跑通一次诊断后才会真正理解的能力。首页可以提示，但不能铺开。

### P2：保留但后置，作为研究/学习产品线

- 拆书图谱
- 样本研究
- 历史任务
- 研究资料

这些适合归入研究工作区。它们是第二产品线，不是首页主任务。

### 不继续独立推广

- 证据对照独立入口
- 评分报告独立入口
- 爆款套路库独立入口
- 研究库问答主入口
- 多书对比首页入口
- 通用账号/模板能力

这些能力如果有价值，应作为结果态、子视图或高级能力存在。

## 页面位置预留

### 首页 / 快速诊断

首屏：

- 左下角模型连接状态，模型卡内保留 AI 设置快捷按钮。
- 痛点标题。
- 快速诊断输入框。
- 载入示例。
- 开始诊断。

左侧导航：

- 常驻：诊断、我的书籍、研究、设置、深度质检。
- 不展示功能归属分组。
- 复诊记录、方法论库、诊断看板、导出资产在进入我的书籍后出现。
- 拆书图谱、样本研究、历史任务在进入研究工作区后出现。
- 证据对照、评分报告只在结果生成后出现。

不再放在首页主区：

- 快速诊断、深度质检、导入整本书籍、AI 设置四个功能卡。
- “功能去向和成熟度”能力地图。
- 已经属于我的书籍或研究工作区的二级功能展示卡。

### 我的书籍 / 项目工作区

应该承接：

- 章节诊断页；
- 批注与意见状态；
- 版本和复诊记录；
- 方法论库；
- 诊断看板；
- 项目导出。
- 故事体检（剧情漏洞、时间冲突、人设一致性、结构图、人物弧光、对话分析）作为当前作品的编辑事实层。

这里可以密集，但首页不能密集。

### 研究工作区

应该承接：

- 整书导入；
- 拆书导览；
- 关键关系故事线；
- 完整图谱探索；
- 样本研究；
- 历史任务；
- 研究资料。

默认先让用户读懂一本书，再进入完整图谱和导出。

### 设置工作区

只承接：

- AI provider；
- 模型配置；
- API Key / Base URL；
- 本地隐私和连接测试。

诊断看板和历史任务不应长期放在设置里；旧路径可以兼容。

## 后续设计验收

- 首页第一眼仍然是快速诊断，不是功能大厅。
- 结果主 CTA 能创建/保存真实改稿版本，不是无行为按钮或只改本地状态。
- 复诊比较 issue 与证据状态，缓存命中不会创建伪版本。
- 指标结论区分事实、候选、模型复核和人工决定。
- 每个保留功能都有明确所属工作区。
- 每个进阶功能都有“何时使用”的说明。
- 旧入口不再扩大曝光。
- 导出和图谱不堆格式，先按用户任务分组。
- 看板不是指标墙，而是复诊教练。
- 方法论库不是抽象概念，而是“下次写前检查规则”。
