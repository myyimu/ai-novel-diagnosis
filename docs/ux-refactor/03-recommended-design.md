# 前端 UX 重构：推荐设计方案

日期：2026-07-10

## 设计结论

推荐采用“任务驱动主工作区 + 条件式上下文检查器”的中等重构方案。

这不是视觉层面的三栏调优，而是把页面从“所有内容同时出现”改成“当前任务优先”。左侧负责定位当前任务，中间负责完成当前任务，右侧只在用户选中具体对象时展示补充详情。全局设置、历史、帮助不再常驻右栏。

## 三种方案对比

| 方案 | 用户体验收益 | 技术改动范围 | 风险 | 对业务逻辑影响 | 响应式复杂度 | 迁移成本 | 回滚难度 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 方案一：保守调整 | 解决最明显的三栏拥挤、重复和移动端崩坏；旧流程基本不变 | 小。主要调整 `ThreeColumnWorkspaceShell`、`WorkspaceLayout`、默认布局开关和部分 Tab 可见性 | 低。不会重塑路由与状态 | 基本无影响；仍使用现有 `workspace-store` 和 `useWorkspaceHandlers` | 中。需要给现有三栏补断点和折叠规则 | 低 | 很容易。可回到经典布局 |
| 方案二：中等 IA 重构（推荐） | 主路径清晰；输入、运行、结果、详情各自有边界；桌面保留效率，移动端可用 | 中。新增页面外壳契约、任务页布局、条件式检查器和路由映射测试 | 中。需要避免拆散现有状态和 handler | 不改变业务规则，只重排 UI 归属；需要加少量 UI 状态 facade | 中。响应式规则可一次建立，多页复用 | 中 | 容易。可按路由或 feature flag 回退 |
| 方案三：彻底页面/路由拆分 | 长期信息架构最干净；结果、历史、项目、整书任务可分享和回访 | 大。需要资源化路由、状态切片、可能的持久化/API 契约调整 | 高。当前 store/handler 强耦合，缺少覆盖完整流程的 E2E | 有间接影响。虽然不改业务含义，但会触碰数据生命周期和缓存假设 | 低到中。最终页面更简单，但迁移期复杂 | 高 | 较难。涉及多路由和数据归属迁移 |

## 方案一：保守调整

### 做法

- 经典布局继续作为默认安全基线。
- 三栏仅作为桌面增强，建议限制在 `>=1280px` 且用户明确开启。
- 右栏没有当前对象时默认隐藏，不显示“帮助/AI 设置/历史任务”等全局 Tab。
- 中栏 Tab 不再固定展示“输入/诊断/结果/分析”四个大类；至少先让不同路由只显示相关 Tab。
- `WorkspaceLayout` 增加断点：平板左栏折叠，移动端单列。
- 修复 resizer 的键盘可达性和左右侧栏 aria label 重复问题。

### 适合场景

适合短期止血，尤其是还没有时间重整路由和状态时。它可以快速避免三栏在移动端和窄屏上的严重退化。

### 不足

它没有解决根因：路由、Tab、右栏和旧 `activeView` 仍是多套层级并存。页面会比现在更可用，但认知结构仍偏混杂。

## 方案二：中等 IA 重构（推荐）

### 核心结构

```text
WorkspaceHeader
└─ WorkspaceTaskFrame
   ├─ TaskNav：当前工作区任务导航
   ├─ TaskMain：当前路由唯一主任务
   └─ ContextInspector：选中对象的详情，可关闭，可降级为抽屉
```

### 职责定义

| 区域 | 唯一职责 | 不应承载 |
| --- | --- | --- |
| 左侧 | 当前工作区内的任务入口，例如快速诊断、深度质检、整书研究 | 全局设置、结果详情、帮助说明、可操作表单 |
| 中间 | 当前路由的主工作流，例如输入章节、运行诊断、查看报告 | 多个不相关任务、全局配置、长期历史列表 |
| 右侧 | 当前选中对象的上下文详情，例如证据、历史条目、方法论卡、图谱节点 | 永久在线的历史/设置/帮助/项目范围 |

### 右侧显示条件

右侧检查器只在以下条件满足时显示：

1. 用户选中了可查看详情的对象；
2. 当前对象的详情会帮助完成主任务；
3. 关闭右侧后不会丢失主任务进度；
4. 移动端可用 bottom sheet 或详情页表达同一信息。

没有选中对象时，右侧不应占位。全局设置、模型服务、帮助、历史任务应通过路由、弹窗或抽屉进入，不常驻右栏。

### 推荐路由归属

| 路由/能力 | 推荐主结构 | 说明 |
| --- | --- | --- |
| `/diagnose/quick` | 单一快速诊断任务页 | 输入态、运行态、结果态是同一路径的状态，不拆成横向 Tab |
| `/diagnose/deep` | 步骤页 | 参考资料 → 生成标准 → 评分；依赖顺序明确 |
| `/diagnose/score` | 兼容路由或深度质检结果视角 | 当前与 `/diagnose/deep` 共享 `chapter` view，应先明确真实差异 |
| `/diagnose/evidence` | 结果详情或证据视角 | 更适合从报告中的证据点击进入检查器或详情页 |
| `/project/current` | 项目上下文/概览 | 只保留项目选择、当前资产和最近运行状态 |
| `/project/revisions` | 改稿方案列表 | 与快速诊断结果相关，但可独立回访 |
| `/project/methodology` | 方法论库 | 独立内容库，不放在快速诊断主路径首屏 |
| `/project/export` | 导出页 | 低频操作，独立页面或结果页次级操作 |
| `/research/book` | 整书拆解任务 | 上传 → 预览 → 任务进度 → 结果 |
| `/research/compare` | 样本对比 | 独立研究任务 |
| `/research/patterns` | 模式/图谱视角 | 依赖整书结果，适合对象详情检查器 |
| `/research/materials` | 研究资料库 | 独立列表页 |
| `/settings/provider` | 全局 AI 设置 | 独立设置页，不在右栏常驻 |
| `/settings/input` | 输入偏好/高级默认值 | 设置页或诊断页高级设置 |
| `/settings/ai-settings` | 与 provider 合并或保留兼容入口 | 当前命名与 provider 重复，需要统一 |

### 快速诊断目标体验

```text
空状态
→ 填写章节与必要条件
→ 点击诊断
→ 显示运行状态
→ 报告成为主内容
→ 输入区折叠为“本次稿件摘要”
→ 主 CTA：复制改稿 Prompt / 重新诊断
→ 点击证据或历史时打开右侧检查器
```

这样保留现有完整报告内容，但结果出现后不再让输入区继续占据主视觉焦点。

### 状态设计

不建议立刻拆 `workspace-store`。推荐先增加 UI 层 facade：

- 业务状态仍由 `workspace-store.ts` 和 `use-workspace-handlers.ts` 管理；
- 布局状态独立为 route-scoped UI store，例如当前任务、检查器对象、折叠状态；
- 旧 `activeView` 作为兼容层逐步退场；
- 路由参数和页面状态的映射必须由测试锁定。

这能避免在信息架构重构初期同时改业务逻辑和 UI 结构。

## 方案三：彻底页面和路由拆分

### 目标结构

```text
/diagnoses/new
/diagnoses/:runId/summary
/diagnoses/:runId/evidence
/diagnoses/:runId/revisions
/projects/:projectId
/books/new
/books/:jobId/overview
/books/:jobId/graph
/settings/provider
```

每个资源有稳定 URL，历史结果、证据、改稿方案和整书任务都可以直接回访。

### 适合场景

适合产品已经确认诊断结果、复诊资产、整书任务都要长期保存并分享时。它能显著改善长期维护性。

### 当前不推荐作为第一阶段的原因

- 现有 `use-workspace-handlers.ts` 同时承载导航、API、缓存和派生状态，直接拆路由会牵动太多业务假设。
- 现有测试没有覆盖路由映射、响应式和完整用户流。
- 当前首要问题是认知负担和布局职责，不是数据模型必须立即资源化。
- 一次性拆分会让回滚成本变高。

## 最终推荐

选择方案二。

原因是它对当前项目的约束最匹配：

- 可以保留现有业务规则、API 调用和缓存含义；
- 可以先迁移一个完整代表流程，即“快速诊断”；
- 可以在经典布局和旧三栏之间保留回退路径；
- 可以逐步补测试，再迁移深度质检、项目、研究和设置；
- 可以解决用户路径、三栏职责和响应式问题，而不是只调整视觉。

方案一适合作为临时兜底，但不能消除多套层级并存。方案三长期更理想，但现在会把 UI IA、路由、状态和数据生命周期同时打开，风险不匹配。

## 推荐设计细则

### 主视觉焦点

桌面端主视觉焦点始终是中间主工作区。左侧导航保持稳定但低权重；右侧检查器只有在用户主动选中详情时出现。

### 响应式规则

| 视口 | 布局 |
| --- | --- |
| `>=1280px` | 左侧任务导航 + 主工作区 + 条件式右侧检查器 |
| `768px-1279px` | 左侧可折叠或抽屉；主工作区占主宽度；右侧作为 overlay inspector |
| `<768px` | 单列；顶部/底部导航；详情以 bottom sheet 或独立页面出现 |

### 无障碍要求

- 一个页面只有一个主 `h1`。
- 左侧任务导航使用明确的 `aria-current="page"`。
- Tab 必须具备 `tablist`、`tab`、`tabpanel`、`aria-controls`、键盘方向键切换。
- 可调整宽度的分隔条必须可键盘操作，或在重构中取消鼠标专用 resizer。
- 右侧检查器打开后，焦点移动到标题或第一个可操作控件；关闭后回到触发元素。
- 移动端 bottom sheet 必须有焦点限制和 Escape/关闭按钮。

### 明确不做

- 不引入新的 UI 框架。
- 不修改诊断算法、请求参数含义、缓存命中规则或历史数据语义。
- 不把所有信息都强塞进三栏。
- 不把设置、帮助、历史任务作为常驻右栏内容。
