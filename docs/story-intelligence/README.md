# 故事体检与叙事智能产品设计

> 本设计服从 [`../product-doctrine.md`](../product-doctrine.md)。六项能力不是新的“AI 评分器”，而是把整书编辑的事实核对、结构检查和修改复盘交给作者的证据层。

> 实施任务以 [`implementation-plan.md`](./implementation-plan.md) 和 [`execution-plan.yaml`](./execution-plan.yaml) 为准；当前文档提供产品边界，不代表功能已经全部实现。

## 1. 决策结论

这些能力在能进入编辑取舍与改稿闭环时合理且有价值，但不应被包装成六个并列的“质量分”。建议把它们统一为 **故事体检（Story Audit）**：先把整本小说转成可追溯的场景、事件、人物状态和原文证据，再按编辑标准生成统计、冲突候选和结构图，由作者确认后转成修改任务。

推荐决策如下：

| 能力 | 是否做 | 产品形态 | 首次优先级 | 关键原因 |
| --- | --- | --- | --- | --- |
| 对话占比分析 | 做 | 确定性统计 | P0 | 成本低、可复核，可帮助编辑定位叙述形态；比例本身不判断好坏 |
| 时间冲突 | 做 | 约束检查 + 模型复核 | P0 | 事实边界相对清楚，适合成为一致性能力的第一站 |
| 剧情结构图表 | 做 | 场景/章节/剧情线可视化 | P1 | 帮作者和编辑看清信息与修改影响范围，不以模板偏离扣分 |
| 人设一致性 | 做 | 人物状态账本 + 冲突候选 | P1 | 用户价值高，但必须区分成长、伪装、不可靠叙述和真正跑偏 |
| 剧情漏洞 | 做 | 候选问题工作台 | P1 | 价值高、误报风险也最高，不能直接输出“确定漏洞” |
| 人物弧光 | 做 | 多维变化曲线 | P2 | 帮编辑检查目标、能动性、信念、代价和选择的变化，不能只画情绪折线 |

第一版不应承诺“自动找出所有漏洞”。产品文案应使用：

```text
发现可能影响读者理解的冲突、断线和未兑现线索，并提供原文证据供作者确认。
```

## 2. 为什么不做成一个指标大屏

六项能力属于三种完全不同的信号：

1. **确定性统计**：对话字符占比、对话段落占比、章节长度、人物出场次数。
2. **结构化推导**：时间约束冲突、人物状态变化、伏笔是否仍未回收、剧情线是否长期中断。
3. **编辑性判断**：某个因果缺口是否真是剧情漏洞、某次行为是否违背人物动机、结构偏离是否真的伤害阅读体验。

如果把三类信号混成 0～100 分，用户无法判断结果来自文本计算、模型推断还是编辑偏好。故事体检必须固定展示这条链路：

```text
编辑标准 -> 事实/统计 -> 冲突候选 -> 双证据 -> 可能解释 -> 阅读风险假设 -> 作者确认 -> 修改任务 -> 真实版本复诊
```

## 3. 外部成熟实现带来的设计约束

本设计不以单一审稿方法为依据，主要参考以下成熟产品与研究：

- [Plottr](https://plottr.com/features/) 把章节、剧情线、人物弧和场景卡放在同一时间线，并支持按人物、地点和属性过滤。可借鉴的是“同一场景表，多种视图”，不是照搬拖拽编辑器。
- [Fictionary](https://fictionary.co/support/help-docs/evaluate/) 先逐场景记录人物、剧情和设定元素，再由 Story Map 汇总可视化；其编辑流程也明确覆盖 plot holes、continuity、character arc 和 story arc。可借鉴的是“逐场景事实先于全书结论”。
- [Campfire Timeline](https://campfirewriting.com/timeline-maker) 允许事件、人物弧、日历和多条时间线互相链接。可借鉴的是“叙事顺序”和“故事内时间”必须分开。
- [Sudowrite Characters](https://docs.sudowrite.com/using-sudowrite/1ow1qkGqof9rtcyGnrWUBS/characters/a7tdE1ZB8KvAwMD3Mopwpd) 使用结构化人物卡约束后续大纲、场景和正文生成。可借鉴的是把人物卡视为可更新的项目事实，而不是一次性模型摘要。
- [ProWritingAid Dialogue Report](https://cdn.prowritingaid.com/ebook/ProWritingAid_EBook11.pdf) 和 [AutoCrit Dialogue](https://www.autocrit.com/editing/support/dialogue/) 都把对话识别、占比和原文定位结合起来。可借鉴的是统计可点击回正文，同时避免把通用区间当作硬性质量标准。
- [FACTTRACK](https://arxiv.org/abs/2407.16347) 用带时间的世界状态追踪提升故事大纲矛盾检测；[FlawedFictions](https://arxiv.org/abs/2504.11900) 表明模型的漏洞检测会随故事变长显著退化；[全局时间图研究](https://aclanthology.org/2025.emnlp-main.1601/) 说明孤立判断事件对会造成全局不一致。由此确定本产品必须采用分段抽取、全局约束和二次复核，而不是把整本书一次塞给模型。

## 4. 产品信息架构

### 4.1 不增加新的顶层工作区

继续使用现有四工作区。新增能力归属如下：

| 位置 | 职责 | 展示内容 |
| --- | --- | --- |
| `/diagnose/quick` | 单章急诊 | 只显示本章对话占比、可在本章确认的局部异常；不声称完成全书一致性检查 |
| `/research/book` | 整本导入与预处理 | 上传、拆章、覆盖率检查、启动整书体检；任务完成后进入项目页 |
| `/project/health`（新增） | 当前书籍的故事体检主页面 | 总览、一致性、结构、人物、文本统计五个页签 |
| `/project/current?chapter=...` | 章节证据复核 | 从体检问题跳回原文，显示成对证据和人物/事件状态 |
| `/project/revisions` | 修改与复诊 | 保存 V1/V2 正文、实际采用项，并记录问题是已修复、保留为作者意图、误报、复发还是待处理 |
| `/project/export` | 交付 | 导出 Markdown 与 JSON 故事体检报告 |
| `/settings/dashboard` | 跨任务趋势 | 只放覆盖率、人工采纳/误报、问题解决/复发和无法判断趋势，不放某一本书的结构图或质量总分 |

`/project/health` 应成为“我的书籍”下的二级入口，排序建议为：

```text
书籍与章节 -> 故事体检 -> 正文版本与复诊 -> 经确认的问题和规则 -> 导出
```

### 4.2 故事体检页面

页面顶部先回答三个问题：

- 分析覆盖到哪些章节，是否为部分结果。
- 当前最值得作者确认的 1～3 个问题是什么。
- 这些结论中有多少是统计、规则命中、模型复核或人工确认。

主内容使用五个页签：

1. **总览**：高优问题、分析覆盖率、未确认候选、剧情线健康度。
2. **一致性**：时间、人物事实、人物知识、地点、世界规则、因果链。
3. **结构**：章节/场景卡、主线与支线、关键节拍、伏笔—回收、节奏曲线。
4. **人物**：人物状态账本、目标距离、能动性、信念和关系变化。
5. **文本统计**：对话占比、叙述占比、对话轮次、对白标签、章节分布。

右侧检查器固定展示：对应编辑标准、问题解释、两侧证据、其他可能解释、证据充分度、作者操作。作者操作只有：

```text
确认问题 | 标记为创作意图 | 证据不足 | 稍后处理 | 加入修改计划
```

### 4.3 与当前产品能力的融合映射

故事体检不是第二套整书分析。现有 `BookAnalysisResult` 已经包含大量上游资产，新增能力必须优先复用：

| 当前已有资产 | 当前位置 | 故事体检如何复用 | 只补什么 |
| --- | --- | --- | --- |
| TXT 清洗、章节切分 | `TextPreprocessorService`、`/analysis/book/preprocess` | 作为唯一正文标准化与章节 ID 来源 | 增加场景级切分结果，不再另建预处理器 |
| 异步 Map-Reduce、断点恢复、部分结果 | `BookAnalysisJobService`、`/analysis/book/jobs` | 继续作为唯一整书任务系统 | 任务输入增加 `purpose/profiles`，结果可选带 `storyAudit` |
| `chapterMaps` | `BookAnalysisResult.mapReduce` | 作为场景、事件、人物事实抽取的 Map 基础 | 补 `scenes/events/facts/dialogueAtoms`，保持旧字段兼容 |
| `sourceAnchors`、`chunkEvidenceIndex`、证据搜索 | `mapReduce`、`GET /analysis/book/jobs/:jobId/search` | 作为 finding 跳转和证据回查基础 | 补双证据关联，不再实现第二套全文检索 |
| `characters`、`relationships`、图谱质量 | `BookAnalysisResult`、关系图谱工作台 | 作为实体消歧和人物账本种子 | 增加随场景变化的状态，不复制人物卡和关系图 |
| `plotlines`、`chronicle` | `BookAnalysisResult` | 作为剧情线矩阵与事件时间线种子 | 补故事内时间关系和约束，不另造静态时间线 |
| `chapterFunctionTable` | `writingSupport` | 作为结构图的章节级回退数据 | 有场景表时下钻到场景，无场景表时显示章节级精度 |
| `foreshadowingLedger` | `writingSupport` | 作为 setup-payoff 初始账本 | 增加 reminder、unknown 和证据边 |
| `emotionalBeatMap`、`pacingCurve` | `writingSupport` | 继续负责情绪和节奏视图 | 不把它们伪装成人物弧光 |
| `continuationPack.oocGuards/settingGuards` | `writingSupport` | 作为待作者确认的 canon 候选 | canon 确认后才提高冲突检测权重 |
| `characterVoiceGuide`、`styleBible.dialogueRules` | `generationAssets` | 作为角色语气参照 | 对话占比仍由程序计算，语气漂移后置复核 |
| `RevisionSession` | Web 项目状态 | 现状只承载诊断历史；升级后关联持久化的正文版本链 | 增加 `fromVersionId/toVersionId`、实际采用项、`storyAuditFindingIds` 和问题状态变化 |
| `MethodologyCard` | `/project/methodology` | 只沉淀重复且人工确认的问题 | 误报、创作意图和未确认候选不得入库 |
| `BookExportService` | `/analysis/book/jobs/:jobId/export` | 继续作为唯一整书导出入口 | 增加 audit Markdown/JSON 分区，不新建下载系统 |

建议对现有结果做兼容式扩展，不创建平行缓存：

```ts
interface BookAnalysisResult {
  // 现有字段保持不变
  analysisPurpose?: "own-draft" | "reference-study";
  storyAudit?: StoryAuditResult;
}

interface WorkspaceProject {
  // 现有字段保持不变
  bookJobId?: string;
  analysisPurpose?: "own-draft" | "reference-study";
}

interface RevisionSession {
  // 现有字段保持不变
  storyAuditFindingIds?: string[];
}
```

这样现有 `CachedBookAnalysis` 会自然缓存故事体检，不需要 `CachedStoryAudit`；已有整书任务历史、恢复、搜索和导出仍然有效。

### 4.4 两类整书导入必须分流

当前“整书拆解”同时可能接收作者自己的稿子和成熟参考作品。两者目标不同，导入页应新增一个明确选择：

```text
这是：我的作品 | 参考作品
```

- **我的作品**：默认 profiles 为 `statistics + continuity + structure + character`；结果进入 `/project/health`；不主推“原创化改写资产”。
- **参考作品**：保持当前拆书、关系图谱、结构学习、原创化边界和研究库流程；默认不跑漏洞警报，避免把学习样本当待修改稿。
- 用户可以之后追加 profile，但系统不应同时生成两套重复人物/时间线资产。

### 4.5 与现有核心闭环的连接

融合后的用户路径应是：

```text
快速诊断一章
  -> 保存到当前书籍
  -> 章节增多或导入整本
  -> 复用整书 Map-Reduce 生成故事体检
  -> 在项目页确认 1～3 个问题
  -> 加入修改计划
  -> 修改后复诊
  -> 仅把重复且确认的问题沉淀为方法论
```

边界保持如下：

- `QuickReviewResult.issues` 继续回答“这一章为什么没人追”，不新增全书漏洞 category；可以只保存 `relatedStoryAuditFindingIds` 做交叉引用。
- 深度质检/Rubric 继续回答“相对成熟样本标准写得怎样”，故事体检不替代参考样本评分。
- 关系图谱继续回答“谁和谁如何关联”，人物一致性只在图谱之上增加随时间变化的状态和冲突。
- 诊断看板继续做复诊趋势，不重复展示某一本书的时间线、结构图或人物弧。

## 5. 各能力的产品规格

### 5.1 对话占比分析

第一版由程序计算，不调用模型判定比例：

- `dialogue_char_ratio`：引号内有效字符 / 正文有效字符。
- `dialogue_paragraph_ratio`：含对话段落 / 全部正文段落。
- `dialogue_turn_count`：对话轮次。
- `dialogue_tag_count`：可识别的“说/问/喊”等标签数量。
- `unattributed_turn_count`：连续多人对话中疑似无法定位说话者的轮次，仅标记候选。
- 按章节与滚动 3 章窗口输出，不只给全书平均值。

边界：支持 `“”`、`「」`、`『』` 和常见半角引号；配置项允许排除短信、系统提示、书信和内心独白。题材参考区间只能在有真实参考样本时显示，默认不输出“低于多少就不好”。

### 5.2 时间冲突

先抽取事件，再做约束检查：

- 故事内时间：绝对日期、相对时间、年龄、持续时长、昼夜、季节。
- 叙事顺序：章节与场景在正文中的出现顺序。
- 事件关系：`before`、`after`、`overlaps`、`during`、`same_time`、`unknown`。
- 参与者与地点：用于识别同一时间出现在互斥地点等候选。

规则引擎先发现周期、负时长、年龄不可能、旅行时间不足和互斥地点；模型只负责解释语义和排除倒叙、预言、梦境、转述等情况。

### 5.3 剧情结构图表

底层只保留一份场景表，派生四种视图：

- 章节 × 剧情线矩阵：每条主线、支线、感情线、人物弧在哪些章节推进。
- 事件时间线：叙事顺序与故事内时间可切换。
- 节拍分布：按三幕、英雄之旅、起承转合或用户自定义模板映射。
- 伏笔—回收图：显示 setup、reminder、payoff、abandoned、unknown。

结构模板是对照工具，不是唯一正确答案。没有选择模板时只展示事实结构，不扣分。

### 5.4 人设一致性

为主要人物建立随时间变化的状态账本：

- 稳定事实：身份、年龄、外貌、能力边界、重要经历。
- 可变状态：目标、阵营、关系、持有物、伤势、地点、已知信息。
- 行为倾向：压力下的选择、禁忌、说话方式、价值排序。
- 作者设定：用户手工确认的 canon，优先级高于模型抽取。

发现冲突时必须先判断它属于：事实矛盾、知识越界、能力越界、动机断裂、语气漂移、关系跳变，还是合理成长/伪装/情境变化。

### 5.5 剧情漏洞

剧情漏洞不设置单一分数。第一版聚焦六类可证据化候选：

- 因果缺口：结果发生但缺少必要原因。
- 目标断线：重要目标建立后长期无行动、无放弃说明。
- 线索断线：高权重伏笔没有回收或转化。
- 信息越权：人物使用尚未获得的信息。
- 规则破坏：世界规则只为当前情节临时失效。
- 可行性冲突：距离、资源、伤势、能力或身份不足以支持行为。

每个候选必须包含至少两个证据锚点或一个证据锚点加一条已确认 canon。缺少双侧证据时只能标记 `needs_human`，不能标记 `verified`。

### 5.6 人物弧光

不使用单一“成长值”，而是按场景追踪：

- `goal_distance`：离外部目标更近、持平或更远。
- `agency`：主动选择程度。
- `belief_state`：核心错误信念/新信念的变化。
- `relationship_state`：关键关系的信任、权力和亲密变化。
- `cost_and_choice`：付出的代价和做出的不可逆选择。

图表必须能点击到导致变化的场景。连续中性场景、连续无代价胜利或突然跳变只作为编辑提示，不自动判定为错误。

## 6. 建议顺手纳入的后续指标

等底层场景表稳定后，可低成本派生：

- POV 连续性与头跳候选。
- 场景目标/冲突/转折/出口钩子覆盖。
- 伏笔回收周期与剧情线静默章节数。
- 主要人物出场分布与命名人物引入负担。
- 地点连续性与移动耗时。
- 情绪强度、信息密度和行动/反应场景交替。

这些指标也必须遵守“统计不等于质量、模板不等于规则”。

## 7. 分阶段路线

### Milestone A：可信事实层（P0）

- 新建独立 `StoryAuditResult`，不要继续膨胀 `QuickReviewResult`。
- 复用整书拆章、异步任务和证据锚点。
- 增加场景、事件、人物事实和 canon 数据结构。
- 上线对话占比、分析覆盖率和 JSON/Markdown 导出。

### Milestone B：时间与结构（P0/P1）

- 建立全局事件图和时间约束检查。
- 上线时间冲突候选、事件时间线、章节 × 剧情线矩阵。
- 所有问题支持跳回双侧原文。

### Milestone C：人物与漏洞（P1）

- 建立人物状态账本和知识边界。
- 使用“候选发现 → 独立复核 → 规则校验”流水线。
- 上线作者确认、创作意图、误报和修改计划状态。

### Milestone D：人物弧与复诊（P2）

- 生成多维人物弧光图。
- 对比修改前后问题状态，不只对比分数。
- 把反复确认的问题沉淀到项目方法论，不把一次误报写入长期规则。

## 8. 成功标准与停止条件

首版成功不看“生成了多少指标”，看以下结果：

- 100% 的高优问题有可点击原文证据。
- 统计字段可由相同输入稳定复算。
- 部分分析明确显示覆盖章节，不伪装成全书结论。
- 人工确认集上，时间冲突高优候选精确率达到 85%，人物/剧情漏洞达到 70% 后才默认展示为警报。
- 用户能把问题标记为创作意图或误报，且复诊时保留该决定。
- 单次体检先给 1～3 个最重要问题，不用红点淹没作者。

如果双证据覆盖率不足、模型无法稳定区分倒叙与时间冲突，或误报率连续两个版本高于目标，应暂停扩展指标，优先修复事实抽取和验证层。

## 9. 非目标

- 不预测平台流量或签约概率。
- 不用一个总分替代编辑判断。
- 不把偏离三幕式、英雄之旅等模板视为错误。
- 不自动改写整本小说。
- 不把用户原稿发送到未明确选择的外部模型。
- 不在单章输入上声称完成整书一致性检查。
