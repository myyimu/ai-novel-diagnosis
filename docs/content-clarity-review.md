# AI网文诊断台内容易懂性评审

日期：2026-06-25

## 评审结论

当前产品方向是成立的，但表达方式有一个明显风险：越接近核心能力，文案越像产品经理、研究员和工程师在互相解释系统，而不是在对网文作者说话。

README 前半段已经比较清楚，能讲明白“先诊断为什么没人追，再生成改稿 Prompt”。但从主要能力、Dashboard、方法论库、研究库、深度质检和架构文档开始，术语密度明显升高，容易让用户产生三个误解：

- 这是一个很复杂的写作研究系统，不是一个马上能用的诊断工具。
- 我需要先懂 Prompt、归因、Gate、Rubric、证据链，才能开始使用。
- 产品在强调系统能力，而不是直接告诉我下一版稿子怎么改。

下一阶段不应该继续增加概念，而应该把复杂能力翻译成作者能立刻理解的动作。

## 评审团意见

### 1. 资深网文编辑

作者最关心的不是“归因系统是否完整”，而是：

```text
我这章哪里劝退？
为什么读者不想追？
下一版先改哪一刀？
```

“证据链、结构化 issue、Gate 判断”这些概念可以存在，但不应该站在用户第一层。用户第一层应该看到的是：

- 最大劝退点。
- 原文哪里暴露了这个问题。
- 读者会怎么反应。
- 下一版先改什么。
- 改完后怎么判断是否变好。

### 2. 网文作者代表

目前有些词会让我停下来想它是什么意思，例如：

- Prompt 有效率归因。
- 项目级归因校准。
- 结构化 issue。
- Gate 分布。
- 可迁移 Rubric。
- 研究库就绪度。
- 证据锚点。
- 方法论沉淀。

这些词不是不能用，而是需要换成更像写作现场的话：

- “这条 Prompt 有没有帮上忙？”
- “问题到底是稿子没写出来，还是上一条指令没说清？”
- “这稿该继续改、重写开头，还是先别投？”
- “下次写之前先检查什么？”

### 3. AI 产品经理

产品真正的壁垒是判断和复诊，不是术语。

复杂概念应该变成后台结构，前台只保留结果和动作。比如：

```text
后台：prompt_attribution.calibration.dominantCategory
前台：这次变好主要因为上一条 Prompt 说清了冲突。
```

```text
后台：gateDecision = restructure
前台：建议重写开头，不要只局部润色。
```

```text
后台：methodologyCard.type = prompt_rule
前台：下次让 AI 写开头前，先把主角损失、反击目标、章末钩子写进 Prompt。
```

### 4. 新手体验设计师

现在的第一层体验已经比之前收敛，但还存在“解释太多”的倾向。用户第一次打开时不应该看到太多系统名词。

建议首页和核心结果区遵守一个规则：

```text
一屏只回答一个问题。
```

输入前回答：

```text
把第一章贴进来，我帮你找最大劝退点。
```

输出后回答：

```text
这一版先改这里。
```

复诊后回答：

```text
这次修改有没有用。
```

### 5. 工程负责人

内部结构不需要为了易懂而删除。`Gate`、`Rubric`、`Prompt attribution`、`evidence anchor` 这些都可以继续作为数据模型和代码概念存在。

但用户可见文案需要分层：

- 对作者：用“劝退点、原文证据、下一版动作、改完复诊”。
- 对团队：用“问题类型、处理建议、版本变化、Prompt 是否有效”。
- 对开发者：文档里保留 “Gate / Rubric / attribution / schema”。

也就是说，不是砍能力，而是换翻译层。

## 主要问题

### 问题一：README 后半段从用户语言切回系统语言

比较清楚的表达：

```text
别急着让 AI 重写，先诊断这稿为什么没人追。
```

不够清楚的表达：

```text
诊断看板会展示常见问题、复诊改善趋势、Gate 分布和 Prompt 有效率归因，并给出编辑建议、项目级归因校准、置信度、诊断理由、信号、待补数据和可复制的模型/编辑复核提示。
```

建议改成：

```text
诊断看板会告诉你：最近几版有没有变好，哪些问题总是重复，上一条 Prompt 有没有帮上忙，以及下一版最该补什么信息。
```

### 问题二：方法论这个词有价值，但太硬

“方法论库”对团队和长期用户有价值，但对新手偏抽象。第一层可以换成：

```text
我的改稿规则
下次写前检查
常犯问题
可复用 Prompt
```

保留“方法论库”作为页面名也可以，但页面内要解释成：

```text
这里保存你反复踩过的坑，下一章写之前先看一眼。
```

### 问题三：Dashboard 文案偏分析后台

当前用户可见词包括：

- Prompt 有效率。
- Prompt 有效率推断。
- 归因校准。
- 模型/编辑复核提示。
- Gate 分布。
- 问题类型。

建议页面改成更接近作者动作：

| 当前文案 | 建议文案 |
| --- | --- |
| 诊断 Dashboard | 诊断看板 |
| Prompt 有效率 | Prompt 有没有帮上忙 |
| Prompt 有效率推断 | 上一条 Prompt 是否有效 |
| 归因校准 | 为什么会变好/变差 |
| 模型/编辑复核提示 | 复制给编辑或模型复查 |
| Gate 分布 | 稿件处理建议 |
| 质量趋势 | 最近几版变化 |
| 高频方法论卡 | 常犯问题和改稿规则 |

### 问题四：研究库和深度质检过早暴露专业感

研究库页面目前像“创作决策库”和“可追溯样本池”，专业但不够直觉。

用户更容易理解的入口是：

```text
学习成熟作品怎么留住读者
```

然后再解释：

```text
系统会把样本拆成卖点、人物关系、节奏和证据，帮你对比自己的稿子。
```

深度质检也不建议一开始讲 Rubric。可以先讲：

```text
用成熟样本当参照，看看你的第一章差在哪。
```

Rubric 作为结果中的“评分标准”出现即可。

### 问题五：部分概念同时出现中文和英文，增加理解负担

目前用户可见层同时出现：

- Prompt。
- Rubric。
- Gate。
- Dashboard。
- Map-Reduce。
- quickScore。
- golden fixture。

建议规则：

- `Prompt` 可以保留，因为 AI 写作者已经熟悉。
- `Rubric` 前台改为“评分标准”，开发文档保留 Rubric。
- `Gate` 前台改为“处理建议”，开发文档保留 Gate。
- `Dashboard` 前台改为“看板”。
- `Map-Reduce` 只放技术文档，前台改为“分章拆解”。
- `quickScore` 前台改为“本次诊断分”。
- `golden fixture` 只放开发文档，前台改为“示例报告”或“测试样本”。

## 文案分层原则

### 第一层：作者能马上行动

面向首页、诊断结果、复诊结果。

只使用这类词：

- 第一章。
- 最大劝退点。
- 原文证据。
- 读者反应。
- 下一版先改。
- 改稿 Prompt。
- 改完复诊。
- 常犯问题。

### 第二层：团队能做判断

面向诊断看板、复诊历史、批量结果。

可以使用这类词：

- 问题类型。
- 严重程度。
- 处理建议。
- Prompt 是否有效。
- 最近几版变化。
- 重复问题。
- 待补信息。

### 第三层：开发和专业用户能复核

面向架构文档、API、导出 JSON、内部说明。

可以保留：

- Gate。
- Rubric。
- attribution。
- evidence anchor。
- schema。
- calibration。
- golden fixture。

## 建议优先级

### P0：先改用户第一眼能看到的词

- README “主要能力”中的高密度术语段落。
- 首页和诊断结果区的“证据链、产品诊断、方法论卡片”等标题。
- Dashboard 页面中的 Prompt 归因和 Gate 相关文案。
- 方法论库空状态和卡片标题。

### P1：再改进阶页面入口

- 深度质检：从“Rubric/评分系统”改成“用成熟样本找差距”。
- 样本研究：从“研究库目标/持久化研究库”改成“学习成熟作品怎么留住读者”。
- 拆书图谱：从“图谱资产/证据回链”改成“把人物关系和伏笔看清楚”。

### P2：最后整理开发文档

- 保留专业术语，但增加“用户可见翻译”小节。
- 架构文档中明确哪些词只用于内部，不直接暴露给作者。
- 导出报告里增加“普通版”和“技术版”两种表达。

## 改写示例

### 示例一：Prompt 归因

当前：

```text
Prompt 归因已经抽到共享 ai-core 引擎并输出解释信号和项目级校准建议，但仍需要真实案例校准，不能替代编辑判断。
```

建议：

```text
系统会观察每次改稿后有没有变好，帮你判断上一条 Prompt 是真的有效，还是只是碰巧分数变化。这个判断仍需要你结合正文确认。
```

### 示例二：方法论库

当前：

```text
方法论库不是历史任务列表，而是作者下次改稿前能复用的自查系统。
```

建议：

```text
这里不是单纯保存历史记录，而是保存你反复踩过的坑。下次写开头前，先按这些规则检查一遍。
```

### 示例三：Gate

当前：

```text
Gate 只表示当前稿件的改稿优先级建议，例如继续、修改、重构或废稿；它不是平台流量预测。
```

建议：

```text
系统会给出处理建议：继续写、局部修改、重写开头，或者先别投。它不是流量预测，只是帮你决定下一步怎么处理这版稿子。
```

### 示例四：研究库

当前：

```text
这里不是通用聊天知识库，而是小说创作决策库：所有结论都要回到上传资料、章节证据和对比样本，最终服务选题、卖点重组和第一条高质量提示词（Prompt）。
```

建议：

```text
这里用来学习成熟作品为什么能留住读者。系统会把样本里的卖点、人物关系、节奏和原文证据整理出来，帮你改自己的题材和开头。
```

## 下一步执行清单

1. 建立一张“术语翻译表”，所有用户可见文案先过一遍。
2. 改 README 后半段，尤其是“主要能力”和“当前限制”。
3. 改 Dashboard 页面，把归因、校准、Gate 改成作者能理解的说法。
4. 改方法论库页面，把“方法论”解释为“下次写前检查规则”。
5. 改研究库和深度质检入口，降低第一屏专业感。
6. 保留内部术语在代码和架构文档中，但不要直接推到作者面前。

## 本轮决策

当前不是产品方向错误，而是表达层级需要重构。

产品应该继续保留“诊断、复诊、Prompt 迭代、作者经验沉淀”这条主线，但对用户说话时要更像一个懂网文的编辑：

```text
少讲系统怎么判断，多讲这章哪里劝退、为什么劝退、下一版先改什么。
```
