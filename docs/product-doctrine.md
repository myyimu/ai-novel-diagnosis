# AI网文诊断台产品总原则

本文档是所有产品、工程和 AI 开发文档的最高产品约束。若其他文档与本文冲突，以本文为准。

## 1. 核心方向

AI网文诊断台不是“一键写小说”工具，也不是泛泛评分器。它的核心方向是：

```text
蒸馏编辑判断 -> 指导作者理解问题 -> 给出可执行改稿意见 -> 通过真实改稿复诊验证
```

产品首先服务作者，而不是替作者做决定。AI 的职责是把编辑经验拆成证据、判断依据和行动建议，让作者更清楚地知道下一版该改哪里、为什么改、怎么验证改得是否有效。

## 2. 最短闭环

所有功能都必须能回到这条链路：

```text
输入稿件
-> 找出影响阅读的核心问题
-> 引用正文证据
-> 解释可能的读者反应
-> 给出修改优先级
-> 生成可交给写作 AI 或作者自己的改稿指令
-> 保存真实修改版本
-> 独立复诊问题是否改善
-> 沉淀可复用写作规则
```

不能进入这条链路的能力，只能作为辅助信息或研究功能，不能放在产品第一注意力里。

## 3. 评价边界

系统可以说：

- 这段正文可能让读者误解什么。
- 哪些信息缺少铺垫、回收或因果连接。
- 哪些角色行为与已出现事实存在冲突。
- 哪个修改动作最可能降低理解成本或增强追读理由。
- 这条建议来自统计、规则、模型复核还是人工确认。

系统不能说：

- 这本书一定会火或一定扑。
- 某个分数等同于客观质量。
- 模型发现的剧情漏洞一定成立。
- 工程 fixture 通过就说明产品有效。
- 缓存命中或同模型自评可以证明改稿真的改善。

## 4. 证据优先级

用户可见结论必须遵循证据链：

```text
编辑标准 -> 正文证据 -> 结构化事实 -> 候选判断 -> 替代解释 -> 修改动作 -> 复诊结果
```

不同信号不能混成一个总分：

- 确定性统计：如对话占比、章节长度、证据覆盖率。
- 规则候选：如时间环、同时异地、知识先用后得。
- 编辑判断：如动机是否成立、结构是否伤害阅读体验。
- 作者决定：确认问题、创作意图、证据不足、误报、计划修改、已解决。

## 5. 作者主权

作者可以保留创作意图。系统必须允许用户把 finding 标记为：

- 确认问题
- 创作意图
- 证据不足
- 误报
- 加入修改计划
- 已解决

人工决定必须独立保存，不能写回不可变模型结果里。

## 6. 快速诊断与故事体检的关系

快速诊断解决单章急诊：

```text
这一章为什么可能没人追，下一版先改哪里。
```

故事体检解决整书事实层：

```text
整本书有哪些可追溯的统计、冲突候选、结构断点和人物状态变化。
```

两者共用证据和复诊闭环，但不能混淆语义：

- 单章快速诊断不得声称完成全书一致性检查。
- 整书故事体检不得变成质量总分大屏。
- 深度质检/参考样本用于学习和对照，不替代作者作品的改稿判断。

## 7. 文档维护规则

新增或修改文档时必须检查：

1. 是否仍服务“编辑指导作者改稿”。
2. 是否把模型输出和作者决定分开。
3. 是否把工程验证和产品有效性验证分开。
4. 是否避免把旧的功能大厅、指标大屏、泛研究库重新带回首页主线。
5. 是否给后续 AI 明确的执行边界和验收标准。

过时的产品定位、UX 改造和一次性评审文档应合并到当前权威文档后删除，避免后续 AI 读取到相互冲突的方向。
