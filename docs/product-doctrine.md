# 产品主线、科学边界与交付原则

本文档是全仓产品、设计、模型、数据和研发文档的最高层产品约束。其他文档描述具体功能或历史状态；如出现歧义，以本文为准，并应修正文档而不是保留两套说法。

## 唯一核心方向

AI 网文诊断台不是作品质量裁判、流量预测器，也不是一键代写工具。它要把成熟编辑的工作过程蒸馏成作者可反复使用的写作教练：

```text
编辑标准
-> 文本信号
-> 可回查证据
-> 编辑假设与替代解释
-> 作者意图确认
-> 修改优先级和边界
-> 可执行改稿方案或 Prompt
-> 保存真实新版本
-> 独立复诊
-> 经确认的方法沉淀
```

产品交付的不是“AI 说得对”，而是一个可检查、可选择、可执行、可验证的编辑决策过程。作者始终是最终取舍者。

## 用户价值

| 用户 | 核心帮助 | 不应承诺 |
| --- | --- | --- |
| 新手作者 | 获得接近编辑初筛的检查顺序、证据和第一步改法 | 替代长期写作训练 |
| 有经验作者 | 降低自查盲区，管理整书一致性和版本复盘 | 定义作品审美或唯一正确写法 |
| AI 辅助写作者 | 把初稿问题转成更精确的改稿约束，并验证 Prompt 是否执行 | 由同一模型自写自评就证明有效 |
| 编辑、工作室 | 批量完成初筛、证据整理和问题归档，把时间留给高价值判断 | 自动替代签稿、商业判断和资深编辑终审 |

业务价值按可信度排序：

1. 高可行：证据化初筛、编辑流程标准化、改稿任务生成、版本复盘。
2. 中等可行：优先问题识别、题材化编辑建议、跨章节一致性候选。
3. 低可行且不得包装成确定能力：平台流量、签约概率、读者留存因果和作品绝对质量预测。

## 功能进入产品的共同门槛

剧情漏洞、时间冲突、人设一致性、结构图、人物弧光、对话分析、Rubric 和样本研究都只是编辑教练的证据或工具，不是独立卖点终点。每项能力必须回答：

1. 它对应编辑流程中的哪一个判断？
2. 结论能否回到原文、章节、场景或作者设定？
3. 是否同时提供替代解释和“不确定/需人工确认”状态？
4. 能否转成有边界的改稿动作，而不是泛泛评价？
5. 修改后能否通过真实版本复诊验证？
6. 哪些确认数据能沉淀为通用、题材或作者个人规则？

不能进入这条链路的指标只可作为辅助展示，不能成为首页主任务、总分组成或成功指标。

## 核心产品闭环

快速质检是入口，不是完整产品。完整链路必须保存下列对象：

```text
V1 原稿
-> 结构化 issue（证据、严重度、编辑理由、替代解释）
-> 作者/编辑确认：接受、保留为创作意图、误报、暂缓
-> 修改计划（目标、保留项、禁止项、复诊检查点）
-> V2 真实正文版本
-> 独立复诊
-> issue 变化：解决、改善、未变、复发、新增、无法判断
-> 人工确认本轮经验是否值得沉淀
```

仅切换页面、改变本地状态、重复展示缓存结果或比较两个启发式分数，都不算完成复诊。方法论卡也不能由一次模型输出自动写入长期规则。

## 编辑能力如何蒸馏

优先蒸馏编辑过程，而不是收集漂亮的最终点评。知识资产分为五类：

- 检查标准：编辑在什么阶段检查什么。
- 判断规则：哪些信号支持问题，哪些情况构成例外。
- 证据模式：最少需要哪些原文、上下文、时间或设定证据。
- 修改策略：针对不同根因的最小修复、结构调整和重写边界。
- 复诊标准：什么变化才算解决，什么只是换一种表述。

规则按三层组织：通用编辑方法、题材/平台情境方法、作者个人方法。低层规则不得覆盖高层事实；个人方法只有在作者确认或多轮验证后才能沉淀。

实现顺序默认是：结构化规则与程序统计 -> 检索增强的编辑知识 -> 候选检测与独立复核 -> 人工反馈闭环 -> 有足够干净数据后再做排序、偏好学习或微调。不得把微调当成第一步。

## 科学有效性边界

当前模型输出应称为“有证据的编辑假设”或“风险候选”，不能称为客观质量测量。界面和文档统一遵守：

- 正文引文存在，只能证明引用未伪造，不能证明问题解释或读者影响必然成立。
- `quickScore` 是问题严重度的兼容摘要，不是经标定的作品质量分。
- `confidence` 是证据完整度/上下文充分度提示，不是预测正确概率，未校准前不得显示成百分比准确率。
- 模型推断的“读者影响”使用“可能、风险、假设”等措辞；没有行为数据时不做因果归因。
- 同一模型诊断、改写、复诊会产生自我偏好，关键结论应由不同提示角色、不同模型/规则复核或人工抽检。
- 部分章节、未完结文本和覆盖不足的整书结果必须显式降级，不得推断缺失情节。

## 有效性评测门禁

工程测试只能证明契约和流程可运行，不能证明编辑判断有效。对外声称“有效”前至少完成：

1. 240–400 个中文网文章节或片段，覆盖题材、章节位置、人工/AI 初稿和无严重问题负例。
2. 一部分样本采用受控注入缺陷，另一部分来自真实改稿版本。
3. 每个样本由至少 3 名独立编辑盲标主问题、类别、严重度、证据范围和建议动作。
4. 训练/规则开发、校准和隐藏测试集严格分离。
5. 报告 Top-1 主问题一致率、macro-F1、严重问题误报/漏报、语义证据精度、复测稳定性、跨模型漂移、Brier/ECE 校准。
6. 用原稿、产品指导改稿和普通 Prompt/无工具改稿做盲评对照；只有产品组的目标读者/编辑偏好提升且置信区间高于零，才宣称改稿有效。

上线前允许使用的产品表述是“提供有证据的编辑建议，帮助作者更系统地改稿”；不得使用“准确预测爆款”“客观评分”“保证提升留存”等表述。

这些边界不是保守措辞，而是来自现有研究限制：

- [Help Me Write a Story（ACL 2025）](https://aclanthology.org/2025.acl-long.1254/) 显示 LLM 写作反馈可以具体且多数时候有帮助，但常漏掉最关键问题，也可能在“应该批评还是肯定”的主判断上失败。
- [LitBench（EACL 2026）](https://aclanthology.org/2026.eacl-long.362/) 报告最强现成评判模型与人类偏好的一致率仍约为 73%，不足以充当唯一编辑裁判。
- [G-Eval（EMNLP 2023）](https://aclanthology.org/2023.emnlp-main.153/) 说明 LLM 评估可与人类相关，但相关不等于对单篇稿件的正确概率，且自动评判可能偏好模型生成文本。
- [Judging LLM-as-a-Judge（MT-Bench）](https://arxiv.org/abs/2306.05685) 记录位置、冗长和自我偏好等评判偏差，因此同模型自写自评不能作为效果证明。
- [FlawedFictions](https://arxiv.org/abs/2504.11900) 表明剧情漏洞检测会随故事长度增加而退化；[FACTTRACK](https://aclanthology.org/2025.naacl-long.144/) 支持用原子事实、时间有效性和世界状态替代一次性自由审稿。

因此，产品可行的技术路线是“规则/事实底座 + 模型候选与复核 + 作者/编辑决定 + 真实改稿结果”，而不是依赖单一模型总评。

## 信息架构与优先级

- 首页/快速诊断：完成低门槛编辑初筛，突出证据、首要问题、修改边界和进入真实改稿的 CTA。
- 项目工作区：保存章节正文、版本、作者判断、改稿计划、复诊和个人方法，是核心价值承载区。
- 故事体检：位于项目内，为整书一致性和结构编辑提供事实层；不能另起一套任务、证据或复诊系统。
- 深度质检：用参考样本和 Rubric 支持进阶判断，但评分必须退居证据和行动之后。
- 研究工作区：从有权使用的成熟样本中提炼可迁移方法，服务编辑知识库，不服务内容复制。
- 看板：展示问题解决率、人工采纳/误报、覆盖率和复诊结果，不以总分增长制造伪进步。

## 文档与研发验收

任何 AI 或开发者实施功能时，至少检查：

- 是否写明对应的用户和编辑任务。
- 是否区分事实、规则候选、模型复核和人工决定。
- 是否保存真实版本与作者意图。
- 是否有从 issue 到修改计划再到复诊的端到端测试。
- 是否记录误报、拒绝和无法判断，而不只记录采纳案例。
- 是否避免流量因果、绝对分数和替代编辑的承诺。
- 是否说明产品有效性指标与工程成功指标的区别。
